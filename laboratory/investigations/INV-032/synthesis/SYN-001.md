## Pattern Identification

### Pattern 1: Multi-Process Browser Architecture

**Evidence**: Electron and Chromium use multi-process architecture with separate browser/main, renderer, GPU, and utility processes that separate privileges and fault boundaries [1][3].

**Synthesis**: Modern desktop runtimes universally adopt browser-based multi-process models where:
- Main process handles native APIs and window management
- Renderer processes host web content with varying privilege levels
- IPC mechanisms enable secure communication between processes
- Sandboxing provides defense-in-depth

### Pattern 2: IPC Performance Trade-offs

**Evidence**: IPC round-trip latency measured in 0.1-0.6ms ranges for Electron; Tauri v2 adds raw payload and Channel API to address JSON serialization overhead for large payloads [2][5][6].

**Synthesis**: IPC is essential for browser-to-backend communication but introduces overhead:
- JSON serialization adds latency for large payloads
- Binary transfer (ArrayBuffer, channels) reduces overhead
- Structured Clone Algorithm limits serializable types
- Synchronous IPC blocks and should be avoided

### Pattern 3: System WebView vs Bundled Browser

**Evidence**: Tauri using system WebView2/WebKitGTK/WKWebView shows order-of-magnitude smaller bundles and faster cold starts vs Electron bundling Chromium [2][4].

**Synthesis**: Two architectural choices exist:
- **Bundled browser** (Electron): Larger binary, consistent behavior across OS, complete control
- **System webview** (Tauri): Smaller binary, faster startup, OS-dependent behavior

### Pattern 4: SQLite as Default Embedded Database

**Evidence**: SQLite is ACID-compliant, public-domain, with language bindings for all major platforms and supports WAL mode for concurrent reads [7].

**Synthesis**: For desktop applications requiring local persistence:
- SQLite is the default choice for embedded databases
- WAL mode enables concurrent readers, single writer
- CRDT-based sync layers enable offline-first multi-device
- Encryption available via SEE or SQLCipher

### Pattern 5: Cross-Platform Packaging Spectrum

**Evidence**: electron-builder supports NSIS (consumer), MSI (enterprise), DMG (macOS), AppImage (Linux) with signing requirements per platform [8].

**Synthesis**: Desktop packaging requires platform-specific decisions:
- Consumer apps: NSIS/DMG/AppImage with auto-update
- Enterprise: MSI/PKG with MDM/deployment tools
- Auto-update: NSIS and AppImage supported; MSI not supported

### Pattern 6: Industrial Air-Gapped Deployment

**Evidence**: Windows IoT Enterprise supports offline activation via ePKEA; Linux uses Yocto/Buildroot with signed boot images and OSTree for transactional updates [9].

**Synthesis**: Industrial deployment requires:
- Reproducible builds with signing
- Offline activation mechanisms
- Transactional updates with rollback
- Provisioning packages for factory deployment

## Architectural Knowledge

### Knowledge 1: Desktop Runtime Selection Criteria

| Criterion | Electron | Tauri | Native |
|-----------|----------|-------|--------|
| Binary Size | Large (bundled Chromium) | Small (system webview) | Varies |
| Startup Time | Slower | Faster | Fastest |
| Cross-Platform | Excellent | Good | Requires ports |
| Native Access | Via IPC | Via Rust | Direct |
| Performance | Good | Excellent (Rust) | Excellent |
| Bundle Weight | 100-200MB | 5-20MB | Variable |

### Knowledge 2: IPC Best Practices

1. **Prefer async patterns**: `ipcRenderer.invoke` + `ipcMain.handle` over synchronous `sendSync`
2. **Minimize round-trips**: Batch operations where possible
3. **Use binary for large data**: ArrayBuffer, MessagePort for bulk transfers
4. **Enable contextIsolation**: Protect renderer from malicious content
5. **Expose minimal APIs**: via contextBridge with narrow permissions

### Knowledge 3: Security Model Components

| Component | Purpose | Implementation |
|-----------|---------|----------------|
| Context Isolation | Separate preload from page JS | `contextIsolation: true` |
| Sandbox | OS-level process isolation | `sandbox: true` |
| Node Integration | Node.js in renderer | `nodeIntegration: false` |
| CSP | Content source restrictions | HTTP header configuration |
| Code Signing | Authenticity verification | Platform-specific signing |

### Knowledge 4: Embedded Database Patterns

| Pattern | Use Case | Implementation |
|---------|----------|----------------|
| Single-file SQLite | Local-first apps | rusqlite, QSQLITE, sqflite |
| WAL mode | Concurrent reads | `PRAGMA journal_mode=WAL` |
| Sync layer | Multi-device | CRDT (SQLite Sync), LiteSync |
| Encrypted | Security | SQLCipher (AES-256) |

### Knowledge 5: Industrial Deployment Checklist

- [ ] Reproducible builds with version control
- [ ] Code signing (platform-specific certificates)
- [ ] Offline activation configured (ePKEA, OEM license)
- [ ] Secure boot enabled (signed boot images)
- [ ] Transactional updates with rollback
- [ ] Provisioning packages for factory deployment
- [ ] Removable media policies
- [ ] Application whitelisting (WDAC/AppLocker)
